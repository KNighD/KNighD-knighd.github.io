从零搭建组件库

发表于 2022-02-24 更新于 2022-04-29 分类于 web前端 阅读次数：
本文字数： 2.8k 阅读时长 ≈ 5 分钟

写一个玩票性质的组件库，单纯为了尝试 pnpm + vite + storybook +jest 等工作中还没机会使用的框架，这里做一个开发过程的记录，想到哪儿写到哪儿，可能比较混乱，持续更新。

初始化项目

  1. 安装 pnpm，官方添加了很多安装方式，这里我挑一个 curl -f https://get.pnpm.io/v6.16.js | node - add --global pnpm
  2. vite 初始化项目 vite create my-components 根据提示选择 react + ts 的模板
  3. 添加 commitlint，规范提交信息，参考文档 https://commitlint.js.org/#/guides-local-setup?id=install-commitlint
  4. 添加 eslint / prettier / editorconfig
  5. 添加 linted-staged 及配置 husky 用于提交前检测 eslint 与 prettier 是否通过。pnpx mrm@2 lint-staged 可以一键安装，需要注意的是确保 eslint 和 prettier 要在 devDependencies 里，参考 issue https://github.com/sapegin/mrm/issues/158

处理完上述步骤，基本的架子就搭好了，做到这里的时候想到虽然 vite 也提供了不少模板，但总有些与自己的风格不符，不如把目前的代码作为模板，将来就不需要重复配置了。通过 github 的 import 功能，轻松复制了一下当前的仓库，等日后有时间再完善成 vite template

起步

文档

文档通过 storybook 展示，storybook 依赖于 webpack，而项目中使用的反而是 vite，那不如直接把 story 和 components 作为不同包，由此想到使用 monorepo。

不过 pnpx sb init 似乎不兼容 pnpm，初始化过程中会通过 npm 安装，初始化之后，删除 node_modules 执行 pnpm install 则会提示，启动 storybook 会报错 Cannot find module '...webpack.config'，见 https://github.com/storybookjs/storybook/issues/13428，由此看来 pnpm 的兼容性还有待研究，由此 pnpm 也被放弃

monorepo

如今 monorepo 的方案有很多，例如 lerna / nx / rushstack，或者直接使用包管理工具 pnpm / yarn 2 / npm 7 等。由于 rush 上手成本比较高, 原计划使用 lerna + pnpm

到这一步想到可能这个仓库就不太简单是个组件库了，不如就作为一个私人的工具库吧，其下面的 components 才是组件库。

在根目录下执行 pnpx lerna init，可以看到目录下多了 packages 和 lerna.json。

修改 lerna.json 及 package.json

// lerna.json
{
  "packages": ["packages/*"],
  "version": "independent",
  "npmClient": "pnpm",
  "useWorkspaces": true
}
// package.json
{
  // ...
  "workspaces": [
    "packages/*"
  ],
}

然后将代码分别移动到 packages 下

在一番腾挪之后发现，lerna bootstrap 无法给 packages 下的包安装依赖,见 https://github.com/prisma/language-tools/issues/272 不过按以下顺序重新走一遍流程又可以生效，比较奇怪

初始化项目 – lerna 初始化 – 到 packages 中 vite 创建 components，cra 创建 stories – lerna bootstrap

不过 lerna 的一些问题也在这个过程中暴露出来，比如长期不维护，老issue，幽灵依赖等，所以我打算尝试一下 nrwl/nx

初始化项目2

执行 npx create-nx-workspace --preset=react

在 13.9.x 版本运行项目会出现提示 https://github.com/nrwl/nx/issues/7924，不过项目的重点不在 apps，而是 libs

添加 ui 库

nx generate @nrwl/react:lib ui-components

为 ui 库添加 storybook

nx generate @nrwl/react:storybook-configuration ui-components

整个项目只有根目录有一个 package.json，那么如果要发布的时候版本号从哪里获取呢？

查阅文档发现 lib 是存在两种模式的 https://nx.dev/structure/buildable-and-publishable-libraries#publishable-and-buildable-nx-libraries，那我们这里应该选择 publish

那么先把先前建立的 ui-components 移除掉

nx g @nrwl/workspace:remove ui-components

发现 ui-components 在 ui-components-e2e 中有依赖，先移除 ui-components-e2e，之后再移除 ui-components

重新添加

nx generate @nrwl/react:lib ui-components --publishable --importPath @rellika/ui-components

同理创建一个 ui-themes

运行 storybook

nx run ui-components:storybook

样式与主题

样式方面，国内大部分选择 scss / less，material-ui 选择 styled-jsx，我这里选择 styled-components

构建

https://github.com/developit/microbundle

Button

为了方便定制，在 Styled 装饰的组件上，都添加了样式名。例如StyledNativeButton 这里就会有一个 RuiButton-nativeBtn 的样式名。命名的规范就是 Rui + 组件名 + 装饰的组件。

文件命名规范：组件使用 AaBb 这样的 Pascal，文件夹使用 aa-bb 这样的 snake

Jest

nx test ui-components

添加 jest-dom 方便对 dom 进行断言

npm install --save-dev @testing-library/jest-dom
