关于 Tree Shaking 的探索

发表于 2022-01-27 更新于 2022-02-14 分类于 web前端 阅读次数：
本文字数： 11k 阅读时长 ≈ 18 分钟

近期项目组的组件库需要维护，其中有关于组件按需加载采用 antd 组件库的方案，是通过 umijs/babel-plugin-import 来实现的。

这个方案实际上是通过分析语法，在编译阶段将模块引用转成通过相对路径来实现的。即

import { Button } from 'antd';
ReactDOM.render(<Button>xxxx</Button>);

      ↓ ↓ ↓ ↓ ↓ ↓

var _button = require('antd/lib/button');
ReactDOM.render(<_button>xxxx</_button>);

然而这个方案实际上早在很久之前已经被 antd 官方抛弃了，antd 目前的是通过 tree shaking 实现 JS 部分的按需加载而， css 方面则认为性价比不高等原因没有额外处理，略有争议，详细可以参考 https://github.com/ant-design/ant-design/issues/23988 这个 issue 里的相关讨论。

鉴于 webpack 里的 tree shaking 比较晦涩，借着尝试将原项目调整为tree shaking 按需加载的契机，我将在 tree shaking 方面的探索的相关知识整理如下。

最早些时候注意到 tree shaking 这个词，应该是对比 rollup 与 webpack 这两种打包工具的异同，那会儿 webpack 应该是还未能支持到 tree shaking 的，也正是基于此，antd 出了 babel plugin import 这个方案。目前来说其实主流的打包工具都应该是支持 tree shaking 的。

tree shaking 是一个术语，通常用于描述移除 JavaScript 上下文中的未引用代码(dead-code)。它依赖于 ES2015 模块语法的 静态结构 特性，例如 import 和 export。这个术语和概念实际上是由 ES2015 模块打包工具 rollup 普及起来的。

由于webpack5 实际上对 tree shaking 做了一些升级，但我们项目中目前还多是 webpack4 ，因此我们还是先从 webpack4 的 tree shaking 开始。

先来解释几个概念

  1. dead code，即未引用的代码
  2. sideEffects，副作用，这个我们经常在函数式编程里听到。在导入时会执行特殊行为的代码，这个行为可能会影响到全局的环境就算是有副作用了，例如 console.log，引入 css 文件等等
  3. tree shaking，即将无副作用的 dead code 移除，仅保留有效的代码，以减少打包后的体积

如何能让 webpack 的 tree shaking 生效呢？

整个大前提是需要使用 ES2015 模块语法（即 import 和 export）。

我们建一个简单的 DEMO 来看看

// 目录树如下
├── node_modules
├── package-lock.json
├── package.json
├── src
│   ├── index.js
│   └── utils.js
└── webpack.config.js

// webpack.config.js
const path = require('path')

module.exports = {
  entry: './src/index.js',
  output: {
    filename: 'bundle.js',
    path: path.resolve(__dirname, 'dist'),
  },
  mode: 'none',
}

// package.json
{
  "name": "tree-shaking-demo",
  "version": "1.0.0",
  "main": "index.js",
  "scripts": {
    "build": "webpack",
  },
  "devDependencies": {
    "webpack": "^4.46.0",
    "webpack-cli": "^4.9.2"
  }
}

注意：这里为了避免 webpack 一些默认配置项的干扰，将 mode 设置为 none

// utils.js
export const add = (a, b) => a + b
export const minus = (a, b) => a - b

// index.js
import { add } from "./utils";
add(1, 2)

可以看到上述的代码中，实际上 minus 就属于 dead code。

如果使用了 import，则 webpack 打上 /* harmony import / 这个标识，如果使用了 export，则会打上 /\ harmony export (binding) */ 这个标识。

usedExports

让我们修改一下 webpack 的配置，将 usedExports 这个属性打开

const path = require('path')

module.exports = {
  entry: './src/index.js',
  output: {
    filename: 'bundle.js',
    path: path.resolve(__dirname, 'dist'),
  },
  mode: 'development',
  optimization: {
    usedExports: true,
  },
}

再次打包后

/***/ "./src/utils.js":
/*!**********************!*\
  !*** ./src/utils.js ***!
  \**********************/
/*! exports provided: add, minus */
/*! exports used: add */
/***/ (function(module, __webpack_exports__, __webpack_require__) {

"use strict";
eval("/* harmony export (binding) */ __webpack_require__.d(__webpack_exports__, \"a\", function() { return add; });\n/* unused harmony export minus */\nconst add = (a, b) => a + b\nconst minus = (a, b) => a - b\n\n//# sourceURL=webpack:///./src/utils.js?");

/***/ })

/******/

对比上一次的打包，可以 minus 的标识变成 unused harmony export minus 了，即说明这个模块被 webpack 识别为无用的导出。

那么如果我们改成 commonjs 是什么效果呢？

/***/ "./src/utils.js":
/*!**********************!*\
  !*** ./src/utils.js ***!
  \**********************/
/*! no static exports found */
/*! exports used: add */
/***/ (function(module, exports) {

eval("module.exports.add = (a, b) => a + b\nmodule.exports.minus = (a, b) => a - b\n\n//# sourceURL=webpack:///./src/utils.js?");

/***/ })

可以看到的是 minus 不再有 unused harmony export minus 的标识了。

这里对应文档提到的第一点前提：需要使用 ESM，因为 webpack 才可以识别到是否模块导出但未被使用。

但是到这里为止，minus 依然被打包进来了，即 webpack 仅做标识，并不进行代码移除。要移除代码，我们需要配合一些压缩插件，这里我们使用官方推荐的 terser

// webpack.config.js
const path = require('path')
const TerserPlugin = require('terser-webpack-plugin')

module.exports = {
  entry: './src/index.js',
  output: {
    filename: 'bundle.js',
    path: path.resolve(__dirname, 'dist'),
  },
  mode: 'none',
  optimization: {
    usedExports: true,
    minimize: true,
    minimizer: [
      new TerserPlugin(),
    ],
  },
}

那么我们再次打包

function (e, t, r) {
    'use strict'
    r.r(t)
    var n = r(1)
    Object(n.a)(1, 2)
  },
  function (e, t, r) {
    'use strict'
    r.d(t, 'a', function () {
      return n
    })
    const n = (e, t) => e + t
  },

可以看到代码虽然压缩了，但是 minus 的代码确实已经移除掉了。

但如果我们将 mode 改为在 development ，则依然保留未使用的代码，应该是因为某些配置项导致的（不太清楚是哪个默认配置）。而在 production 模式下则是默认开启了 tree shaking 的优化。

这里对应文档提到的第二点前提：需要 production 模式（目前为止效果相当于启用了上述 usedExports 和 TerserPlugin）

####sideEffects

usedExports 是 webpack 通过 terser 检测语句来识别副作用的，但有时候 terser 识别出语句存在副作用而打包进来

例如，我希望增强一下 log 这个函数，现在让我们增加一个 log.js 文件

// log.js
function getEnhancedLog() {
  return (result) => console.log('result ', result)
}

export const enhancedLog = getEnhancedLog()

然后在 utils.js 中重新导出

export const add = (a, b) => a + b

const minus = (a, b) => a - b

export * from './log'

那这个时候我们实际上没有使用到 enhancedLog，理所当然的，webpack 会把 enhancedLog 相关的代码移除掉。

但如果 getEnhancedLog 这个函数不纯，那就有可能出现问题了

function getEnhancedLog() {
  console.log = window.alert
  return (result) => console.log('result ', result)
}

再次打包后 console.log = window.alert 这段代码实际上却会被打包进来，我们是否需要真的需要有这段副作用的代码呢？webpack 无法确定我们实际的意图。如果此时我希望如果 enhancedLog 没有被使用到的时候 console.log = window.alert 也不会打包进来该怎么处理？

这个时候就需要用到 sideEffcts。

webpack 中有个配置项 optimization.sideEffects， 默认为 true，这个配置项即去识别 npm 包中的 package.json 配置的副作用，并进行 tree-shaking。

那么现在，我们在 package.json 新增一行

"sideEffects": false

同时将 webpack 的 mode 设置为 production，需要注意的是 ‘mode: none’ + ‘terser’ + ‘usedExport’ 这种配置或者是 mode 为 development 下是无法生效的。

当前的 webpack 配置如下

const path = require('path')
const TerserPlugin = require('terser-webpack-plugin')

module.exports = {
  entry: './src/index.js',
  output: {
    filename: 'bundle.js',
    path: path.resolve(__dirname, 'dist'),
  },
  mode: 'production'
}

此时再次打包，我们发现 console.log = window.alert 这段代码就已经不见了。

sideEffects 这个字段默认为 true，即当 webpack 认为你的代码中存在副作用时，就会将相关代码打包进来。但如果粗暴将其设置为 false，则会产生新的问题，例如我们经常引入 css 文件

// index.css
body {
  background-color: 'red';
}

// index.js
import './index.css'
import { add } from "./utils";

add(1, 2)

// webpack.congfig.js
const path = require('path')

module.exports = {
  entry: './src/index.js',
  output: {
    filename: 'bundle.js',
    path: path.resolve(__dirname, 'dist'),
  },
  mode: 'production',
  module: {
    rules: [
      {
        test: /\.css$/i,
        use: ['style-loader', 'css-loader'],
      },
    ],
  },
}

如果 sideEffects 为 false，我们会发现 css 的样式也一并也移除掉了。

那么我们就需要配置告诉 webpack 哪些文件是存在副作用的，调整 package.json 如下

"sideEffects": ["**/*.css"]

再次打包，由于我们声明了 css 文件是存在副作用的，而其他文件不存在副作用（如 log.js），此时可以看到确实是将 css 代码打包进去了，而 log.js 的代码则没有。

对于开发库来说，我们经常需要重新导出模块，如下

// index.js
export { a, b } from './module1';
export { c, d, e } from './module2';
export { f } from './module3';

此时一定要需要注意配置正确的 sideEffects。

只有当库配置了正确的 sideEffects，业务使用者的 webpack 才会根据其配置进行 tree-shaking。

如果我们能确保库没有额外的副作用，那么可以设置 “sideEffects”: false，例如 vue。

如果存在一些副作用，则也要做对应的声明，如 antd 的设置

"sideEffects": [
  "dist/*",
  "es/**/style/*",
  "lib/**/style/*",
  "*.less"
]

我们可以建一个 react 的 demo，引入 antd，在 node_modules/antd/package.json 里调整 sideEffects 看看打包结果。

如果只是业务开发者，个人认为一般不用去配置 sideEffects，大部分情况下 webpack 的运行结果是符合预期的

/*#PURE*/

可以看到 sideEffects 实际上是在文件层面的副作用声明，比如 css 文件 / log.js 文件等等，但如果我需要更为细粒度的声明呢？

现在让我们把 log.js 里的内容移动到 utils.js 里看看

// utils.js
export const add = (a, b) => a + b

export const minus = (a, b) => a - b

function getEnhancedLog() {
  console.log = window.alert
  return (result) => console.log('result ', result)
}

export const enhancedLog = getEnhancedLog()

此时，不管你在 sideEffects 里声明 utils.js 是否存在副作用都没有作用，这里我直接粗暴地设置为 false ，webpack 都将会把 console.log = window.alert 打包进去

"sideEffects": false

那么我们要怎么告诉 webpack 不需要将 console.log = window.alert 打包进去呢？

很简单，通过 /*#PURE*/ 注释即可

export const add = (a, b) => a + b

export const minus = (a, b) => a - b

function getEnhancedLog() {
  console.log = window.alert
  return (result) => console.log('result ', result)
}

export const enhancedLog = /*#__PURE__*/getEnhancedLog()

这里告诉 webpack 调用 getEnhancedLog 不会有副作用的，请放心删除。再次打包可以发现确实 console.log = window.alert 不见了。

如果同学们平时足够细心可以发现我们写的 react 组件，babel 转译后都会加上这个 /*#PURE*/ 正是基于这个原因。

const App = () => <div />

      ↓ ↓ ↓ ↓ ↓ ↓

"use strict";

var App = function App() {
  return /*#__PURE__*/React.createElement("div", null);
};

总结

以上是在 webpack4 的基础上对 tree shaking 的一个探索与总结，webpack5 对 tree shaking 又新增一些新的特性，例如对 commonjs 的支持等等特性，还有待继续测试探索。
